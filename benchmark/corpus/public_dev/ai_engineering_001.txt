Resilient distributed systems depend on explicit assumptions about failure rather than an expectation of continuous connectivity. Consider a replicated service with nodes N = 5 and a majority quorum q = 3. If two nodes are unavailable, the service can still accept a write because three replicas can agree on an order. The arithmetic is simple, but the engineering consequence is not: a timeout cannot reveal whether a remote operation failed or merely completed after the caller stopped waiting.

An implementation should therefore attach a stable idempotency key to every logical write. Repeating request R with key K must return the result already associated with K instead of creating a second mutation. This rule limits duplicate work during retries, yet it does not solve every consistency problem. A stale reader may still observe an older value, and a partitioned minority must refuse writes even when doing so reduces availability.

These trade-offs are captured by the relationship between consistency, availability, and partition tolerance, but the familiar slogan can hide operational detail. Engineers still need bounded retry budgets, exponential backoff, jitter, circuit breakers, and measurements of queue age. A service-level objective should distinguish user-visible success from internal attempt counts. Otherwise, a retry storm can make the dashboard appear busy while the useful completion rate is falling.

The design recommendation is to make failure semantics part of the public API. Clients should know which operations are idempotent, how long keys remain valid, and which errors are safe to retry. Tests should inject delayed responses, dropped connections, duplicated messages, and clock skew. Reliability emerges from these observable contracts, not from a single algorithm or a claim that the network will usually behave.
